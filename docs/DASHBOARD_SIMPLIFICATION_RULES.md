# Dashboard Simplification Rules

## Purpose

Defines strict dashboard UX rules for Smart Society.

This file prevents AI agents from generating:
- cluttered admin panels
- analytics-heavy dashboards
- confusing interfaces
- feature overload

The dashboard should feel:
```txt
simple
clear
action-focused
```

especially for:
- elderly users
- non-technical residents
- first-time users

---

# Core Philosophy

Before simplifying, always Deconstruct & Audit the existing Information Architecture (IA) to find and remove "khichdi" (clutter).

The dashboard is NOT:
```txt
a data analytics platform
```

The dashboard IS:
```txt
a task and information hub
```

---

# Most Important Rule

Users should instantly understand:
- what is important
- what needs action
- where to tap

within:
```txt
3–5 seconds
```

---

# Dashboard Priority Order

Dashboards should prioritize:

```txt
1. Important Actions
2. Pending Tasks
3. Important Updates
4. Secondary Information
5. Analytics (minimal)
```

---

# Resident Dashboard Rules

# Resident Dashboard Must Stay Extremely Simple

Residents do NOT need:
- operational analytics
- management charts
- admin metrics
- complex widgets

---

# Recommended Resident Dashboard

## Primary Sections

```txt
Pending Maintenance
Visitor Requests
Latest Notices
Raise Complaint
Quick Actions
```

That is enough.

---

# Maximum Dashboard Density

Avoid:
- too many cards
- too many charts
- too many colors
- excessive statistics

---

# Hard Rule

Resident dashboard should ideally stay under:
```txt
4–6 primary sections
```

---

# Admin Dashboard Rules

Admins can handle:
- more information
- operational visibility
- house hierarchies and relationship mapping (too complex for mobile)

But still avoid:
- chaos
- analytics overload
- widget spam

---

# Card Rules

# Cards Must Have Clear Purpose

Every card should answer:
```txt
Why does this exist?
```

If unclear:
- remove it
- simplify it
- merge it

---

# Good Card Examples

```txt
Pending Visitor Approvals
Pending Complaints
Due Maintenance Payments
```

---

# Bad Card Examples

```txt
Random analytics
Decorative metrics
Meaningless percentages
```

---

# Information Hierarchy Rules

# Important Information Must Visually Stand Out

Examples:
- unpaid maintenance
- emergency notices
- waiting visitor approvals

These should:
- appear higher
- use stronger hierarchy
- remain easy to scan

---

# Secondary Information

Examples:
- historical trends
- old analytics
- inactive stats

Should remain visually lighter.

---

# Action-Oriented Design

Dashboards should focus on:
```txt
what user should do next
```

NOT:
```txt
showing everything possible
```

---

# Quick Action Rules

Important actions should remain:
- obvious
- large enough
- easy to tap

---

# Recommended Quick Actions

Examples:
```txt
Approve Visitor
Pay Maintenance
Raise Complaint
View Notices
```

---

# Analytics Rules

# Analytics Should Stay Minimal

Avoid:
- large complex charts
- too many graphs
- dashboard overload

Most users:
- do not care
- do not understand analytics-heavy UI

---

# Allowed Analytics

Only show:
- actionable insights
- simple summaries
- operational visibility

---

# Example

Good:
```txt
12 complaints pending
```

Bad:
```txt
multi-axis complaint trend graph
```

for normal residents.

---

# Mobile Dashboard Rules

# Mobile Is Primary

Dashboard should:
- scroll naturally
- remain vertically organized
- avoid horizontal layouts

---

# Mobile Priority Sections

Only highest-value information should appear first.

---

# Widget Rules

# Avoid Widget Addiction

Too many widgets create:
- stress
- confusion
- cognitive overload

Every widget must justify its existence.

---

# Empty State Rules

Empty states should:
- reassure users
- explain situation
- suggest next action

---

# Example

```txt
No pending visitors today.
New visitor requests will appear here.
```

---

# Notification Visibility Rules

Important alerts should remain:
- visible
- prioritized
- actionable

Avoid:
- notification clutter
- noisy alert systems

---

# Color Rules

Use colors:
- sparingly
- meaningfully

Avoid:
- rainbow dashboards
- excessive accent colors

---

# Typography Rules

Prioritize:
- readability
- scanning
- calm hierarchy

Avoid:
- tiny metadata-heavy UI
- compressed layouts

---

# Spacing Rules

Dashboards should breathe.

Avoid:
- dense packed sections
- tiny spacing
- cramped cards

---

# Loading Rules

Every dashboard section should support:
- loading state
- empty state
- error state

Avoid:
```txt
blank dashboard sections
```

---

# Elderly-Friendly Rules

# Mandatory Principles

## Reduce Cognitive Load

Users should NOT need:
- tutorials
- explanations
- exploration

---

## Clear Language

Avoid:
```txt
technical system wording
```

Prefer:
```txt
simple action language
```

---

# Example

Avoid:
```txt
Generate Financial Statement
```

Prefer:
```txt
View Society Expenses
```

---

# AI-Agent Dashboard Rules

## Frontend/UI Agents MUST NEVER

### Generate Fancy SaaS Dashboards

This is NOT:
- crypto UI
- startup analytics
- trading dashboard

---

### Add Too Many Metrics

Only important metrics allowed.

---

### Add Complex Charts By Default

Charts should remain:
- optional
- useful
- minimal

---

### Overcrowd Home Screens

Prioritize clarity over density.

---

### Create Multi-Level Dashboard Chaos

Keep navigation shallow and obvious.

---

# Recommended UX Feeling

The dashboard should feel:

```txt
Calm
Simple
Guided
Trustworthy
```

Users should feel:
- comfortable
- informed
- in control

NOT:
- overwhelmed
- confused
- mentally tired

---

# Final Dashboard Philosophy

The best dashboard is NOT:
```txt
the one showing most information
```

The best dashboard is:
```txt
the one helping users complete tasks effortlessly
```
